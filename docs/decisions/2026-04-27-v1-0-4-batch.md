# v1.0.4 batch — CI workflow + .gitattributes + HarnessCard + failure log

Status: Accepted
Date: 2026-04-27
Owner: @inder

## Context

Four independent items from the awesome-harness audit, batched because
each is small and shipping them as separate releases would be churn.

## Decision

### #28 — CI workflow

`.github/workflows/validate.yml` runs `make validate-full` on every PR
and every push to main. Permissions read-only. Single Ubuntu + Python
3.11. Install `pyyaml` + `jsonschema` + `pytest`; run validate-full;
run pytest on `tests/harness/`. Without this, pre-commit hooks are
bypassable with `git commit -n` and the harness becomes opt-in honor
system.

### #29 — `.gitattributes`

Single file at repo root forcing `eol=lf` on every text extension we
use (`.json`, `.yaml`, `.py`, `.sh`, `.md`, `.toml`). Binary
extensions explicitly marked. Without this, Windows checkout converts
LF→CRLF on text files; the determinism gate (`make harness`
byte-identical re-run) breaks.

### #23 — HarnessCard

`.harness/HARNESS_CARD.yaml` declares at-a-glance what this harness
covers, using the CAR (Control / Agency / Runtime) decomposition from
the awesome-harness paper. Includes:
- Counts (29 checks, 20 generators, 10 yamls)
- CAR axes (control, agency, runtime)
- `coverage.covered` + `coverage.not_covered` arrays — the latter
  honestly lists what we deliberately skip (evals, prompt injection,
  sandboxing, long-running checkpoints, etc.)
- Distribution (`bootstrap_command`, `sync_command`, signing key)
- `consumer_fit` profiles (monorepo / Python-only / TS-only / etc.)

Schema at `.harness/schemas/HARNESS_CARD.schema.json` enforces shape;
`harness_policy_schema` validates on every commit. The `version` field
in the card must be bumped in the same commit as `.harness-version`
on every release (manual; mechanism for automation deferred).

### #25 — Rolling failure log

`tools/run_validate.py` now captures stdout from each check, parses
`[ERROR]` lines via the canonical regex, and appends a JSONL entry to
`.harness/.failure-log.jsonl`. Format:

```json
{"ts": "2026-04-27T20:15:01+00:00", "rule": "Q13.route-needs-auth",
 "file": "backend/api/foo.py", "line": 42,
 "commit": "ab12cd34", "host": "Mac.lan", "session": "<uuid>"}
```

- Single writer (the orchestrator); no race.
- Session UUID set once per `run_validate` invocation so all entries
  in one run group together.
- Rotation at 10 MB: live log moves to `.failure-log.jsonl.1`; older
  rotation deleted. No rotation chain beyond `.1`.
- File gitignored (per-machine telemetry).
- Log writes are best-effort — any IO error is swallowed so a disk-
  full condition can never abort `validate-fast`.

Two tests cover the happy path + rotation behavior.

## Consequences

- Positive — CI workflow closes the bypass loophole. Pre-commit becomes
  the inner-loop convenience; CI is the truth.
- Positive — `.gitattributes` eliminates Windows diff churn before it
  shows up.
- Positive — HarnessCard answers "does it cover X?" in 30 seconds for
  any future consumer instead of requiring a 24-file read.
- Positive — failure log gives the AI (and humans) trend visibility:
  "this rule fired 47 times this week" → candidate for tightening or
  refactor target. Closest thing to "evals" that costs $0.
- Negative — `tools/run_validate.py` now captures stdout per check
  instead of streaming it. We re-print verbatim so user-visible output
  is unchanged, but a 1-2 ms latency per check is added (negligible
  against ~18s wall).
- Negative — HarnessCard `version` field is manually maintained.
  Mismatch with `.harness-version` is possible until release ritual is
  documented (RELEASES.md update is part of the bump).
- Neutral — failure log is per-machine, not aggregated. Future work
  could tail multiple machines into a central store; not in scope.

## Alternatives considered

- **CI matrix (multi-OS, multi-Python)** — rejected: standalone harness
  has zero OS-specific code; complexity > value.
- **HarnessCard auto-derived from generator output** — rejected:
  high-value fields (`covered`/`not_covered`/`consumer_fit`) require
  human judgment that no generator can produce honestly.
- **Failure log writes inside `_common.emit()`** — rejected:
  multiple checks running in parallel would race on the file. Single-
  writer in the orchestrator is race-free.
- **Per-check JSONL files merged later** — rejected: complicates
  rotation + the AI consumer reads one path, not N.
