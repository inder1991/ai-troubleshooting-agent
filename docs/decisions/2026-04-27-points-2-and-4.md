# Strip DebugDuck-specific tests from carve + surface loader failures

Status: Accepted
Date: 2026-04-27
Owner: @inder

## Context

Two gaps surfaced by the awesome-harness-engineering audit:

- **Point 2** — the v1.0.0 extraction at `inder1991/ai-harness` has 106
  failing tests because the carve manifest (`tools/extraction/manifest.txt`)
  pulled in DebugDuck-specific test files. Every file under
  `tests/harness/configs/` (12 files) plus `tests/harness/test_directory_claude_mds.py`
  hardcodes DebugDuck paths (`backend/`, `frontend/`, alembic.ini, vitest
  thresholds, etc.) and fails in any other consumer.
- **Point 4** — `tools/_session_start_hook.sh` swallowed
  `tools/load_harness.py` failures with `|| true`. If the loader
  crashed, Claude Code started the session with zero harness context
  AND zero warning.

## Decision

### Point 2 — Manifest exclusion

Add to `tools/extraction/build_manifest.py`:

- `EXCLUDE_TOKENS` gains `/tests/harness/configs/` so every file under
  that directory is skipped during carve.
- New `EXCLUDE_FILES` tuple holds individual project-relative paths to
  drop (initial entry: `tests/harness/test_directory_claude_mds.py`).
- The collector applies both filters before adding any path.

Re-running `python3 tools/extraction/build_manifest.py` regenerates
`tools/extraction/manifest.txt` (366 → 360 entries; 14 dropped, 8 added
since v1.0.0).

### Point 4 — Visible warning on loader failure

Rewrite `tools/_session_start_hook.sh` to:

- Capture loader stdout + stderr into separate temp files (cleaned via
  `trap`).
- Pass loader stdout through verbatim on success.
- On non-zero exit:
  - Emit a multi-line `[HARNESS_WARN]` block to stdout (which is what
    Claude Code consumes) explaining: exit code, that the AI is starting
    without harness context, how to re-run manually, and the first 800
    bytes of loader stderr.
- The hook itself ALWAYS exits 0, so a transient loader bug never aborts
  Claude Code's session start.

Two new tests under `tests/harness/test_session_start_hook.py` (5 → 7):

- `test_wrapper_surfaces_loader_failure` — mock-failing loader (exit 17,
  stderr "boom") → wrapper exit 0, stdout contains `HARNESS_WARN`,
  `exited 17`, and the stderr preview.
- `test_wrapper_passes_through_loader_stdout_on_success` — happy path:
  loader stdout flows through, no `HARNESS_WARN` noise.

## Consequences

- Positive — a fresh `bash tools/extraction/extract.sh` followed by
  `pytest tests/harness/` in `/tmp/ai-harness` should now go from
  106 failing → ~0 failing on harness self-tests.
- Positive — when a consumer's `load_harness.py` crashes, the AI
  immediately sees a structured warning explaining why context is
  missing, instead of silently behaving as if no harness exists.
- Positive — exit code 0 for the hook preserves the safety property that
  a loader bug never breaks Claude Code session start.
- Negative — the warning consumes ~400-500 chars of context window when
  the loader fails. Acceptable; failure should be loud.
- Neutral — `tests/harness/configs/` and `test_directory_claude_mds.py`
  remain in DebugDuck (they're useful here); they just don't ship to
  consumers.

## Operational follow-up (this branch)

Cut a fresh `v1.0.1` on `inder1991/ai-harness`:

1. Re-run `bash tools/extraction/extract.sh` (uses the updated manifest).
2. Smoke-test the extracted repo's `pytest tests/harness/`.
3. Force-push `main` on the standalone repo (no external consumers; v1.0.0
   tag preserved).
4. `git tag -s v1.0.1 -m "..."` (signed; required by the H.3.5 hardening
   from the prior commit) + `git push origin v1.0.1`.
5. Cut a GitHub release pointing at v1.0.1.
6. Bump `.harness-version` here to `v1.0.1`.

## Alternatives considered

- **Move the DebugDuck-specific tests to `backend/tests/` instead of
  excluding them in the manifest** — rejected: the tests genuinely test
  harness setup (mypy/tsc baselines exist, dependencies.yaml shape, etc.),
  they just embed DebugDuck-specific values. Cleanest fix is "don't ship
  them downstream"; rewriting them as fully-generic property tests is a
  larger sprint.
- **Have the hook write the warning to stderr instead of stdout** —
  rejected: Claude Code captures stdout as session context; stderr is
  discarded. The warning has to land in the AI's view to be useful.
