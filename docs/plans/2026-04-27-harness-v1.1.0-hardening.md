# Harness v1.1.0 ‚Äî Production Hardening Plan

**Goal:** Close the 6 P0 bugs from the SDET production-readiness audit and
ship a signed v1.1.0 release. Bumps minor because the on-disk baseline
format changes (breaking).

**Architecture:** Centralize the H-16 emit/parse contract in
`_common.py` so every check, plus `run_validate.py` and
`refresh_baselines.py`, agrees on path shape, line shape, and the regex
that round-trips them. Make path normalization the only place that
knows about `REPO_ROOT`. Migrate-on-read so legacy baselines load.

**Tech Stack:** Python 3.14, `fcntl`, GPG (Ed25519 key
`73A7AF8F04F40EC9`), git filter-repo for source-of-truth republish.

## Stories

### S1 ‚Äî B1: relative baseline paths
- `_common.normalize_path(file)` returns repo-relative POSIX
- `_common.emit()` calls it (with pseudo-path bypass for `gitleaks`,
  `mypy`, `tsc`, `git`, `--target`, anything ending in `/`, anything with
  `://`)
- `_common.load_baseline()` migrates each entry's `file` at READ time;
  drops foreign-machine absolute entries with `[WARN]`
- All 25 in-check `sig = (...)` sites use `normalize_path(...)`
- **Acceptance:** 302/302 harness tests pass; same-machine v1.0.x
  baselines load with zero drops

### S2 ‚Äî B5: single regex source
- `_common.ERROR_LINE_PATTERN` defined once with `.+?` file capture
- `run_validate.py` imports as `ERROR_LINE_RE`
- `refresh_baselines.py` imports as `LINE_RE`
- `output_format_conformance.py` `OUTPUT_LINE_RE` switched to `.+?`
- **Acceptance:** `grep -RI "\\\\S+?.*rule=" .harness tools` returns
  nothing

### S3 ‚Äî B6: escape control chars in emitted messages
- `_common._escape_field(text)` replaces `\n`, `\r\n`, `\r`, `\t`, `"`
- `_common.emit()` runs message + suggestion through it
- **Acceptance:** docstring containing literal `\n` round-trips through
  the H-16 line without splitting

### S4 ‚Äî B2: fcntl lock around failure-log writes
- `tools/run_validate.py` imports `fcntl` (best-effort; falls back on
  Windows)
- `_append_failure_log` takes `LOCK_EX` for the duration of the JSON
  write, releases in `finally`
- **Acceptance:** two parallel `make validate-fast` invocations produce
  N+M valid JSON lines (no interleaving)

### S5 ‚Äî B4: setup_signing.sh defaults to --local
- New `--local` (default) / `--global` / `--force` flags
- Refuses to overwrite an existing `user.signingkey` or `tag.gpgsign`
  without `--force`
- `tests/harness/test_sync_harness.py::_seed_upstream` explicitly sets
  `tag.gpgsign=false` so a developer's global config can't sabotage the
  test run
- **Acceptance:** running the script in a fresh clone does NOT mutate
  `~/.gitconfig`

### S6 ‚Äî B3: init_harness refuses --target == source
- `tools/init_harness.py` resolves both paths and exits 2 with a
  diagnostic if equal
- **Acceptance:** `python3 tools/init_harness.py --target .` exits 2
  with "resolves to the harness source repo itself"

### S7 ‚Äî refresh_baselines --migrate-paths
- New flag walks every `*_baseline.json` in `.harness/baselines/`,
  normalizes paths in place, drops foreign entries with `[WARN]`
- Atomic `.new` ‚Üí `rename` write pattern (same as `_refresh_one`)
- Idempotent
- **Acceptance:** running the migration twice in a row produces
  byte-identical output the second time

### S8 ‚Äî release artifacts
- `tools/init_harness_templates/RELEASES.md` gains a v1.1.0 section
  documenting the breaking change and the one-line migration
- ADR `docs/decisions/2026-04-27-baseline-paths-relative-v1.1.0.md`
- Source-repo work (out of scope for this consumer-side branch):
  `bash tools/sign_release.sh v1.1.0`, push tag, bump consumers'
  `.harness-version` to v1.1.0

## Rollout

1. Merge this branch (S1‚ÄìS7 + docs) into `main` in this consumer repo.
2. In the source `ai-harness` repo: re-run `bash tools/extraction/extract.sh`,
   sign and push tag `v1.1.0`.
3. Per consumer: bump `.harness-version` to `v1.1.0`, then run
   `python3 tools/refresh_baselines.py --migrate-paths` once. Commit the
   migrated baselines.
