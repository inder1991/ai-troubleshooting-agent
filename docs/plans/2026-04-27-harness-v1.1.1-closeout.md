## What B8 surfaced (intended side-effect)

Before v1.1.1, CI ran `make validate-fast`. The plan called for swapping
to `make validate-full`, which adds (a) `FULL_ONLY_CHECKS` (5 heavy
checks), (b) `typecheck_policy` (mypy + tsc), (c) `run_tests` (pytest
against `backend/tests/` + `tests/harness/`).

After the swap, the harness layer is fully green:

```
[VALIDATE] check:output_format_conformance  exited 0 (5.0s)
[VALIDATE] check:backend_testing            exited 0 (4.6s)
[VALIDATE] check:backend_async_correctness  exited 0 (1.0s)
[VALIDATE] check:backend_db_layer           exited 0 (0.9s)
[VALIDATE] check:frontend_testing           exited 0 (0.5s)
... 28/28 checks PASS ...
```

**Consumer-side `backend/tests/` collection fails (exit 2, ~37s).**
Pytest hangs at import time — likely a module-load DB connection or
similar side-effect in a `conftest.py`. This is **not** a regression
from v1.1.1. validate-fast never gated it; v1.0.x and v1.1.0 silently
ignored it. B8's whole point was to expose exactly this kind of issue.

**This consumer repo (ai-troubleshooting-systetm) needs to fix its
backend test collection separately.** Ticket-style summary:

- Repro: `backend/venv/bin/python3 -m pytest backend/tests/ -q`
- Symptom: pytest hangs at collection (>2 min wall) then exits 2.
- Likely cause: `backend/tests/conftest.py` or one of its imports
  triggers a blocking I/O (DB connect, network call) at module load.
- Fix-then-merge: each consumer adopting v1.1.1 must verify
  `backend/tests/` collects cleanly before flipping the CI workflow.

## Decision

The v1.1.1 harness release is sound and ships as planned. The CI
behavior change (B8) requires consumers to fix their own test
collection issues — that's the contract's intended effect, not a
release blocker for the harness substrate.

**Recommended consumer rollout note** (will go into the source-repo
RELEASES.md):

> v1.1.1's B8 fix flips the CI workflow from `validate-fast` to
> `validate-full`. If `backend/tests/` (or `tests/harness/`) was
> previously broken in your repo without anyone noticing, this PR
> will be the first one to fail CI. That's the bug, not the fix —
> diagnose your test collection before merging the v1.1.1 bump.
